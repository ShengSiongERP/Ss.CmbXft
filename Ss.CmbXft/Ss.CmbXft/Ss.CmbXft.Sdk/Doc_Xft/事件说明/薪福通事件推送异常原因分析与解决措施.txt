薪福通事件推送异常原因分析与解决措施
更新时间: 2025-12-10 18:59:53
本文主要介绍薪福通事件推送异常原因分析与解决措施。

异常原因及建议解决方案
超时连接-E01
原因

在薪福通回调应用事件推送地址时，要求在3秒内收到回复，若没有收到回复，将被认为是事件推送失败，错误码为E01。

解决方案

1）系统将自动重试三次，分别间隔1s，2s，4s，均失败，则认为事件异常推送；

2）由于连接超时，在薪福通内认为失败，但在应用方存在成功执行的可能性，请核对应用内业务是否执行成功；

3）若需对事件进行补偿，存在以下几种方案：

重发推送
适用场景包括：在薪福通内无法再次触发、业务发生无需顺序执行等。
要求：应用内处理业务必须支持幂等。且要求对重发事件内容及应用内对事件消费逻辑进行分析。
举例：待办完成发送事件，触发业务创建单据。
风险：如薪福通内依次发送同一员工新增、修改、删除事件，按执行顺序推送。若其中员工修改事件失败，此时最终一致性应达到员工删除状态，若重发员工修改事件，则可能达不到最终一致性，这取决于应用内消费事件的处理逻辑。而在同一个场景中，若其中员工删除事件失败，此时重发员工删除事件，基本上均可达到效果。所以在使用重发推送解决事件异常时，要求对重发事件内容及应用内对事件消费逻辑进行分析，防止出现ABA、非幂等重复处理等情况。
操作限制：由于此操作风险较高，目前只有高权操作可以操作。操作路径：事件日志-失败日志-重发
业务重复执行
适用场景包括：在薪福通内可以通过操作再次触发。
要求：需关注业务最终处理状态。
举例：如员工信息修改事件，可以通过反复编辑保存重新触发事件。
应用业务内直接修改
适用场景包括：应用内处理该业务具备补偿机制。
举例：待办完成发送事件，触发业务创建单据，业务内可直接创建单据，补偿本次失败处理。
风险：需要注意系统间的最终一致性要求。若存在不同操作人员进行不同的补偿措施时，可能导致的重复处理。
特别需要注意的是，在应用内业务处理时，必须满足接口幂等性！

接口返回值不符合规范-E02
原因

在薪福通回调应用事件推送地址收到响应时，薪福通要求返回格式满足，在不满足该结构时，系统将认为事件推送失败，错误码为E02。

{    
    "rtnCod":"200",    
    "errMsg":"xxxx",    
}    
解决方案

修复应用推送代码问题，将返回值修复为符合薪福通要求返回格式。

应用自定义错误类型
原因

在薪福通回调应用事件推送地址收到响应时，薪福通识别返回报文，解析rtnCod是否为200，不为200则认为事件推送失败，错误码为rtnCod中的值。

解决方案

业务处理异常，可根据此错误码定位问题并解决。

系统异常推送强制处理
系统停止推送
在达到应用事件推送异常达到阈值后，系统将暂停推送事件，同时通过薪福通消息及开放平台消息告知。

暂停推送后，开始对推送地址进行10分钟一次的连接测试，连接测试完成或重新编辑事件推送地址后恢复推送。

系统暂停推送阶段内，事件以失败事件形态展示，系统不会自动推送，以免造成业务，可使用超时连接的处理方案解决。

异常事件汇总通知
每日6点、18点对过去12个小时进行事件异常推送进行汇总，结果将通过薪福通消息及开放平台消息告知。
